# AI 转人工客服对接方案

## 一、整体架构

```
用户 (H5/小程序/企微)
    ↓
AI 对话引擎 (aiService)
    ↓ [触发转人工条件]
转接调度层
    ↓
IM 云信在线客服系统 (真人客服)
```

---

## 二、转人工触发条件

| 触发条件 | 说明 |
|---|---|
| **用户主动请求** | 用户发送"转人工"、"找客服"等关键词 |
| **AI 置信度不足** | AI 意图识别的置信度低于阈值（如 < 0.6） |
| **多轮未解决** | 连续 N 轮对话仍未命中意图或未解决问题 |
| **敏感/投诉场景** | 识别到投诉、理赔、纠纷等需人工介入的意图 |
| **异常兜底** | AI 回复异常、接口调用失败等系统异常 |

---

## 三、云信客服路由匹配规则（现有规则）

### 商家侧客服匹配（物流联系商家）

优先级：目的地 > 物流公司

### 物流侧客服匹配（商家联系物流）

优先级：类型 > 商家会员号 > 发货人 > 货主

转接调度层需将 AI 对话中提取到的信息（运单目的地、物流公司、问题类型、商家会员号等）传递给云信系统，让云信按既有规则匹配到正确的客服人员。

## 四、推荐实现方案：服务端消息桥接 + 会话上下文同步

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────┐
│  用户端       │     │  后端服务          │     │  IM云信服务端 │
│ (H5/小程序)  │ ←→  │  (aiService)      │ ←→  │  (云信API)   │
└──────────────┘     └──────────────────┘     └─────────────┘
                           ↑                        ↑
                      AI对话引擎               真人客服(桌面端)
```

### Step 1 — AI 对话阶段

- 用户在 H5/小程序中与 AI 对话
- AI 引擎处理意图识别、多轮对话
- 同时记录完整的对话上下文（消息列表、已识别信息、用户意图）

### Step 2 — 触发转人工

- 满足转人工条件时，AI 引擎向用户发送提示："正在为您转接人工客服，请稍候..."
- 后端调用云信服务端 API 创建/发起会话请求

### Step 3 — 上下文同步（"无缝"的关键）

将 AI 阶段的对话摘要推送给接待的真人客服，包括：

- 用户基本信息（商家会员号、联系方式）
- 用户问题摘要（AI 生成的简要描述）
- 关键业务信息（运单号、目的地、物流公司等）
- 之前的对话记录（最近 N 条）

真人客服在云信客户端看到这些信息后，可以直接继续服务，不需要用户重复描述。

### Step 4 — 消息桥接

- 用户仍在 H5/小程序界面中聊天（体验不中断）
- 后端将用户消息转发到云信 → 真人客服
- 真人客服的回复通过云信 API 回传 → 后端 → 推送到用户端
- 用户感知上就是"同一个对话窗口"，只是回复的人从 AI 变成了真人

### Step 5 — 会话结束/回退

- 人工客服解决后关闭会话
- 系统可选择将用户重新交回 AI（"还有其他问题可以继续问我"）

## 五、需要与云信对接确认的关键问题

| 需确认项 | 说明 |
|---|---|
| **服务端 API** | 云信是否提供服务端 API 用于创建会话、发送消息、接收消息回调？ |
| **Webhook/回调** | 真人客服发送的消息，能否通过 Webhook 回调到后端？ |
| **会话创建 API** | 能否通过 API 主动发起一个客服会话（而非只能通过客户端点击）？ |
| **客服路由 API** | 能否通过 API 传入路由参数（类型、商家会员号等）让云信自动分配客服？ |
| **消息格式** | 支持哪些消息类型（文本、图片、文件）？ |
| **在线状态查询** | 能否查询客服在线状态，转接前判断是否有人可接待？ |

### 备选方案

如果云信不提供完整的服务端 API：

- **方案 A**：生成云信会话链接，引导用户在新窗口中继续（体验稍差但实现简单）
- **方案 B**：使用企业微信客服能力替代云信做真人客服承接（开发计划中本身就有企微对接）

## 六、与开发计划的对应关系

此方案对应开发计划 **阶段二**：

> 人工客服转接对接（IM 系统对接、上下文同步）

**建议**：在阶段一完成 AI 对话引擎后，优先确认云信的 API 能力，这会直接决定阶段二的技术方案走向。如果云信 API 能力有限，可能需要提前调整方案，优先走企业微信客服通道。
